1️⃣ UNDERSTANDING WORKFLOW AUTOMATION
────────────────────────────────────

What are GitHub Actions?
- Native CI/CD platform in GitHub
- Automates software workflows
- Triggered by events (push, pull request, schedule)
- Runs on GitHub-hosted or self-hosted runners
- Integrated with GitHub repositories
- Free tier available

Benefits:
✓ Native GitHub integration
✓ No external platform needed
✓ Easy setup with YAML
✓ Rich marketplace of actions
✓ Secrets management built-in
✓ Status checks for PRs
✓ Matrix strategies for testing
✓ Caching for faster builds


2️⃣ WORKFLOW STRUCTURE
─────────────────────

Workflow Directory:
.github/
└── workflows/
    ├── ci.yml
    ├── deploy.yml
    └── tests.yml

Naming Convention:
- YAML files in .github/workflows/
- Any filename with .yml or .yaml
- Multiple workflows allowed

Basic Workflow Structure:
name: Build and Test                     # Workflow name
on:                                      # Trigger events
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:                                    # Job definitions
  build:
    runs-on: ubuntu-latest              # Runner
    steps:                               # Workflow steps
      - name: Checkout code
        uses: actions/checkout@v3
      
      - name: Set up Java
        uses: actions/setup-java@v3
        with:
          java-version: '17'
      
      - name: Run build
        run: mvn clean package

Complete Example:
name: Java CI/CD Pipeline
on:
  push:
    branches:
      - main
      - develop
  pull_request:
    branches:
      - main
  schedule:
    - cron: '0 2 * * *'  # 2 AM daily

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up JDK
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'
      
      - name: Cache Maven packages
        uses: actions/cache@v3
        with:
          path: ~/.m2/repository
          key: ${{ runner.os }}-maven-${{ hashFiles('**/pom.xml') }}
          restore-keys: |
            ${{ runner.os }}-maven-
      
      - name: Build with Maven
        run: mvn clean verify


3️⃣ WORKFLOW TRIGGERS (ON)
────────────────────────

Trigger Events:

Push Event:
on:
  push:
    branches:
      - main
      - 'release/*'           # Release/* branches
    paths:
      - 'src/**'              # Only if src/ changed
      - 'pom.xml'
    paths-ignore:
      - '**.md'               # Ignore markdown

Pull Request Event:
on:
  pull_request:
    types: [opened, synchronize, reopened]
    branches: [main]
    paths:
      - 'src/**'

Issue Events:
on:
  issues:
    types: [opened, labeled]

Schedule (Cron):
on:
  schedule:
    - cron: '0 0 * * 0'  # Weekly
    - cron: '0 2 * * *'  # Daily at 2 AM

Manual Trigger (Workflow Dispatch):
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment'
        required: true
        default: 'staging'

Release Event:
on:
  release:
    types: [created, published]

Watch Event:
on:
  watch:
    types: [started]


4️⃣ WORKFLOW COMPONENTS
──────────────────────

Jobs:
jobs:
  test:
    runs-on: ubuntu-latest
    
  build:
    runs-on: ubuntu-latest
    needs: test      # Must complete test first
    
  deploy:
    runs-on: ubuntu-latest
    needs: [test, build]

Steps:
steps:
  - name: Step 1
    run: echo "First step"
  
  - name: Step 2
    run: echo "Second step"

Actions (Reusable):
- uses: actions/checkout@v3      # GitHub action
- uses: actions/setup-java@v3    # Official action
- uses: docker/setup-buildx-action@v2  # Community action
- uses: custom-org/custom-action@v1    # Custom action

Runners (Execution Environment):
GitHub-hosted:
- ubuntu-latest (ubuntu-22.04)
- windows-latest
- macos-latest

Self-hosted:
runs-on: self-hosted
runs-on: [self-hosted, linux, x64]

Environment Variables:
env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

steps:
  - run: echo ${{ env.REGISTRY }}
  - run: echo ${{ github.actor }}


5️⃣ SECRETS & ENVIRONMENT VARIABLES
──────────────────────────────────

Secrets Management:
- Settings → Secrets and variables → Actions
- Add secrets (e.g., DOCKER_PASSWORD, API_TOKEN)
- Reference: ${{ secrets.DOCKER_PASSWORD }}

Using Secrets:
env:
  DOCKER_USERNAME: ${{ secrets.DOCKER_USERNAME }}
  DOCKER_PASSWORD: ${{ secrets.DOCKER_PASSWORD }}

steps:
  - name: Docker Login
    run: |
      echo "${{ secrets.DOCKER_PASSWORD }}" | \
      docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin

Organization Secrets:
- Shared across multiple repositories
- Settings → Secrets and variables → Actions (organization level)

Environment Variables:
- Repository-level: Settings → Environments
- Workflow-level: env: in workflow file
- Step-level: env: in specific step

Example:
env:
  GLOBAL_VAR: "value"

steps:
  - name: Use variable
    env:
      STEP_VAR: "step value"
    run: echo $GLOBAL_VAR $STEP_VAR


6️⃣ PRACTICAL WORKFLOW EXAMPLE
──────────────────────────────

Create file: .github/workflows/ci.yml

name: Build and Test CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build:
    name: Build and Test
    runs-on: ubuntu-latest
    
    strategy:
      matrix:
        java-version: [11, 17]
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v3
      
      - name: Set up JDK
        uses: actions/setup-java@v3
        with:
          java-version: ${{ matrix.java-version }}
          distribution: 'temurin'
          cache: maven
      
      - name: Build with Maven
        run: mvn clean package
      
      - name: Run tests
        run: mvn test
      
      - name: Upload coverage
        uses: codecov/codecov-action@v3
        with:
          files: ./target/site/jacoco/jacoco.xml
      
      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v3
        with:
          name: test-results
          path: target/surefire-reports/

Pushing to Repository:
1. Create .github/workflows/ci.yml
2. Commit and push:
   $ git add .github/workflows/ci.yml
   $ git commit -m "Add CI workflow"
   $ git push

3. View results:
   - Go to Actions tab in GitHub
   - Click workflow run
   - Check logs

KEY CONCEPTS:
=============
✓ Workflows = automation files
✓ Triggers = events that start workflows
✓ Jobs = parallel or sequential units
✓ Steps = individual tasks
✓ Actions = reusable components
✓ Runners = execution environment
✓ Secrets = sensitive data
✓ Matrix = testing multiple configurations
✓ Artifacts = build outputs
✓ Caching = performance optimization
